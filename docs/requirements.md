# Requirements

## 1. Users

### Primary user

想创作长篇网络爽文、但不想靠纯手工维护全量设定的作者。

### Secondary user

想做 AI 辅助写作实验、需要一个可控流水线而不是一次性生成器的开发者。

## 2. User stories

1. 作为作者，我希望先有总纲、卷纲和单元纲，这样 AI 不会每章都重新发明故事。
2. 作为作者，我希望有伏笔账本和人物圣经，这样长期创作时不容易遗忘和跑偏。
3. 作为作者，我希望每次只生成当前章节卡，这样正文写作更聚焦。
4. 作为作者，我希望有审校流程，这样能识别明显的 AI 味、节奏问题和收口过满。
5. 作为作者，我希望整个过程是文件化、可版本控制的，这样可以回滚和比较。
6. 作为开发者，我希望后续能接外部 LLM，而不推翻前面的本地流程。

## 3. Functional requirements

### FR-001 项目初始化

系统必须能够创建一个新作品项目目录，并写入基础结构模板。

### FR-002 总控资料管理

系统必须能够保存或生成以下基础资料：

- 总纲
- 世界规则
- 人物圣经
- 伏笔账本
- 卷纲
- 剧情单元表

### FR-003 长线结构规划

系统必须能把目标字数拆解成：

- 章节规模
- 卷规模
- 剧情单元规模

### FR-004 章节卡生成

系统必须能够基于当前进度生成下一章的章节卡，包括：

- 本章目标
- 本章冲突
- 信息释放
- 本章伏笔推进
- 不能透露的内容
- 结尾钩子

### FR-005 进度推进

系统必须允许记录已完成章节的摘要、钩子和阶段推进。

### FR-006 审校支持

系统必须能对章节草稿做基础审校，至少覆盖：

- AI 味
- 重复句式
- 解释过满
- 结尾钩子过弱
- 节奏过匀

### FR-007 人机协同

系统必须支持人工覆盖和修改计划，不强制黑箱自动生成。

### FR-008 后续自动化兼容

系统的状态层必须便于未来接入：

- OpenAI 或其他 LLM
- 自动审校回路
- 检索式上下文装配

## 4. Non-functional requirements

### NFR-001 可追溯

所有关键状态应该可以落盘保存，而不是只存在会话里。

### NFR-002 可读性

输出的文件和模板必须便于作者人工检查。

### NFR-003 可维护性

目录和模块必须足够清晰，方便后续迭代。

### NFR-004 低耦合

核心工作流不应绑定某一个模型平台。

### NFR-005 中文优先

产品首先面向中文网文写作，默认文档与模板应服务中文创作场景。

## 5. Out of scope

当前不纳入强制范围：

- 自动投稿平台对接
- 复杂多人在线协作
- 商业化订阅系统
- 云端账户系统

## 6. Acceptance criteria for MVP

`v1.0.0` 至少需要满足：

1. 用户可以初始化一本书。
2. 用户可以得到总纲、卷纲、单元表、人物表和伏笔账本。
3. 用户可以持续生成章节卡并推进进度。
4. 用户可以审校草稿并得到明确修改建议。
5. 仓库可以通过 GitHub tag/版本说明追踪演进。
