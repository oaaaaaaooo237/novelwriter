# Project Plan

## 1. Project name

Shuangwen Pipeline

副标题：长篇网络爽文写作助手 / 写作工作台

## 2. Background

目标不是让一个模型一次性写完整部长篇，而是构建一个能够持续规划、生成、审校、推进的长篇创作系统。

用户痛点主要集中在：

- AI 容易一章一个小故事，节奏太整齐
- 缺少多卷级别的长期铺垫与回收
- 信息差控制弱，悬念常常提前泄露或直接遗忘
- 人物和世界规则不稳定
- 爽点有，但缺少长期追读感
- 工具如果只靠聊天窗口，很难长期维护一本书

## 3. Product decision

项目当前产品形态确定为：

`本地文件化工作流 + Python 核心引擎 + 桌面 GUI + 后续可接入的 Agent/Prompt 流程`

理由：

- 文件比聊天上下文更适合管理 200-300 章以上内容
- GUI 更适合作者日常使用，而不是反复敲命令
- 本地状态层和核心引擎能保证后续接入 API 时不推翻重来
- 更容易接入 GitHub 做版本归档和过程追踪

## 4. Project goals

### Primary goals

1. 构建长篇爽文创作的结构化工作流。
2. 保证总纲、卷纲、剧情单元、人物、伏笔、章节卡之间有持久状态。
3. 让 AI 以“多角色窄任务”方式参与创作，而不是一口气自由生成。
4. 提供 GUI 工作台，承载创建项目、查看规划和后续写作流程。
5. 支持 `1-1000 万字` 的目标字数配置，并优先优化 `100 万字以下` 的体验。

### Secondary goals

1. 让人工总导演只需要盯关键方向和节奏。
2. 保证仓库可维护、可追踪、可版本化。
3. 保证后续能扩展到不同题材，而不仅限于一个模板。

## 5. Non-goals for early versions

以下内容不作为前几版强制目标：

- 先做复杂 Web 前端
- 直接追求完全自主写完任意超长篇
- 一开始就做多模型编排平台
- 一开始就做云端协作

## 7. Risks

### Risk 1

把重点放到“文笔生成”，忽略结构工程。

应对：

先做状态层和流程层，再做正文自动化。

### Risk 2

GUI 先做太重，导致产品逻辑还没稳定就开始前端消耗。

应对：

先做轻量桌面 GUI，只承载核心流程，不追求复杂界面。

### Risk 3

需求无限膨胀，导致版本推进失控。

应对：

严格按版本边界做，不跨版本塞功能。

## 8. Success criteria

满足以下标准，说明项目方向成立：

1. 能稳定管理一本 `1-1000 万字` 目标小说的结构文件。
2. 对 `100 万字以下` 的项目，能给出比较自然的卷级和章节级建议。
3. 能通过 GUI 创建项目并生成有效的模板骨架。
4. 后续审校环节能抓到明显的 AI 味和节奏问题。
5. 用户能明显减少“重复交代背景”和“来回提醒 AI 不要忘”的成本。
