# Project Plan

## 1. Project name

Shuangwen Pipeline

副标题：长篇网络爽文写作助手 / 写作流水线

## 2. Background

目标不是让一个模型一次性写完整部长篇，而是构建一个能够持续规划、生成、审校、推进的长篇创作系统。

用户痛点主要集中在：

- AI 容易一章一个小故事，节奏太整齐
- 缺少多卷级别的长期铺垫与回收
- 信息差控制弱，悬念常常提前泄露或直接遗忘
- 人物和世界规则不稳定
- 爽点有，但缺少长期追读感

## 3. Product decision

项目第一形态确定为：

`本地文件化工作流 + Python CLI + 可接入的 Agent/Prompt 流程`

理由：

- 文件比聊天上下文更适合管理 200-300 章以上内容
- CLI 更容易先把流程打通，再考虑界面层
- 后续接入 API 和自动化 runner 的成本更低
- 更容易接入 GitHub 做版本归档和过程追踪

## 4. Project goals

### Primary goals

1. 构建长篇爽文创作的结构化工作流。
2. 保证总纲、卷纲、剧情单元、人物、伏笔、章节卡之间有持久状态。
3. 让 AI 以“多角色窄任务”方式参与创作，而不是一口气自由生成。
4. 尽量把写作流程推向半自动，最终接近高自动化。

### Secondary goals

1. 让人工总导演只需要盯关键方向和节奏。
2. 保证仓库可维护、可追踪、可版本化。
3. 保证后续能扩展到不同题材，而不仅限于一个模板。

## 5. Non-goals for early versions

以下内容不作为前几版强制目标：

- 先做复杂 Web 前端
- 直接追求完全自主写完 50 万字
- 一开始就做多模型编排平台
- 一开始就做云端协作

## 6. Milestones

### v0.1.0

项目策划和仓库基建

- 计划说明
- 需求说明
- 任务说明
- README
- 版本归档规范
- GitHub 协作模板

### v0.2.0

项目初始化器

- 创建作品目录
- 生成总纲、卷纲、人物、伏笔等模板
- 固化基础目录结构

### v0.3.0

结构化规划生成

- 生成卷纲
- 生成剧情单元
- 生成伏笔账本
- 生成人物圣经

### v0.4.0

章节流水线

- 生成章节卡
- 记录章节推进
- 记录剧情回收点

### v0.5.0

本地审校器

- AI 味检测
- 节奏均匀化检查
- 结尾钩子检查
- 伏笔/人物一致性检查

### v0.6.0

半自动执行

- 接外部 LLM
- 输出 Agent prompt
- 章节生成与审校串联

### v1.0.0

MVP

- 可稳定支持一本书的长期规划与章节推进
- 人工总导演 + AI 编剧组工作流可闭环

## 7. Risks

### Risk 1

把重点放到“文笔生成”，忽略结构工程。

应对：

先做状态层和流程层，再做正文自动化。

### Risk 2

过早做 UI，导致产品逻辑还没稳定就开始前端消耗。

应对：

先用文件和 CLI 验证工作流。

### Risk 3

需求无限膨胀，导致版本推进失控。

应对：

严格按版本边界做，不跨版本塞功能。

## 8. Success criteria

满足以下标准，说明项目方向成立：

1. 能稳定管理一个 40-50 万字项目的结构文件。
2. 能生成连续有效的卷纲、单元卡和章节卡。
3. 审校环节能抓到明显的 AI 味和节奏问题。
4. 用户能明显减少“重复交代背景”和“来回提醒 AI 不要忘”的成本。
